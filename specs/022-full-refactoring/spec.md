# Feature Specification: 全体のリファクタリング

**Feature Branch**: `022-full-refactoring`
**Created**: 2026-03-28
**Status**: Draft
**Input**: GitHub Issue #7 — 全体のリファクタリング

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 命名規則の統一と型の厳密化 (Priority: P1)

開発者がコードを読む際に、変数名・ファイル名・型定義が一貫しており、IDE 上で型エラーや曖昧な `any` 型が存在しない状態になっている。コードの意図が名前から明確に伝わり、型安全性が担保されている。

**Why this priority**: 命名と型は全コードに横断的に影響し、他のリファクタリング作業の前提となる。ここが揃わないと、構造変更やデータ管理改善の効果が半減する。

**Independent Test**: ビルド時に `any` 型キャストが 0 件であること。同一概念を指す変数名が一貫していること（レビューで確認可能）。

**Acceptance Scenarios**:

1. **Given** リファクタリング前のコード, **When** `any` を含む型キャストを検索する, **Then** 意図的な型キャスト以外の `any` 使用が 0 件である
2. **Given** 同一の概念（例: スプライト URI マップ）を扱う複数ファイル, **When** 型定義を確認する, **Then** 単一の型定義を共有しており、重複定義が存在しない
3. **Given** ファイル名と対応する export 名, **When** 命名パターンを確認する, **Then** ファイル名は kebab-case、export は camelCase/PascalCase で統一されている

---

### User Story 2 - プロジェクト構造と責務の整理 (Priority: P1)

各ファイル・モジュールの責務が明確で、重複コードが排除されている。開発者が新機能を追加する際に、変更すべきファイルが直感的に特定できる。不要なファイルは削除・統合され、ディレクトリ構造が論理的に再編成されている。

**Why this priority**: 構造の見通しが悪いと、全ての変更作業においてコストが増大する。基盤としての改善価値が高い。

**Independent Test**: 重複していた関数（`buildSpriteUriMap` 等）が単一箇所に集約されていること。各ディレクトリの責務がファイル配置から読み取れること。不要ファイルが存在しないこと。

**Acceptance Scenarios**:

1. **Given** 2 箇所に存在していた `buildSpriteUriMap()` 関数, **When** リファクタリング後にコードを検索する, **Then** 単一の共有モジュールにのみ定義されている
2. **Given** `SpriteUriMap` 型が 4 ファイルに重複定義されていた状態, **When** リファクタリング後に型定義を検索する, **Then** 単一箇所からエクスポートされ、他ファイルはインポートしている
3. **Given** ゲームロジック内のマジックナンバー（閾値・レート）, **When** リファクタリング後に確認する, **Then** ゲームプレイ・ビジネスルールに関わる定数が名前付き定数として集約されている（UI スペーシングやアニメーション値は対象外）
4. **Given** プロジェクトのディレクトリ構造, **When** リファクタリング後に確認する, **Then** 各ディレクトリが単一の責務を持ち、不要・未使用ファイルが削除されている

---

### User Story 3 - データ管理の一元化 (Priority: P2)

アプリケーションの状態データがどこに存在し、どうアクセスされるかが明確に集約されている。ローカルコンポーネントで持つべきデータ以外は、React Context Provider を通じて一元管理されている。

**Why this priority**: 現状のデータフローは extension → webview の一方向メッセージングで機能しているが、グローバル変数（`window.__SPRITE_URI_MAP__`）や暗黙的なデータ受け渡しがあり、データの流れの見通しを改善する余地がある。

**Independent Test**: `window` オブジェクトへの直接代入・参照が排除されていること。状態の読み書きが React Context Provider 経由で行われていること。

**Acceptance Scenarios**:

1. **Given** `window.__SPRITE_URI_MAP__` として注入されていたスプライトデータ, **When** リファクタリング後に確認する, **Then** React Context Provider 経由で型安全に受け渡されている
2. **Given** 各 webview コンポーネント, **When** 状態へのアクセスパターンを確認する, **Then** ゲーム状態とスプライトデータは Context/カスタム Hook 経由でアクセスされ、直接的なグローバル変数参照がない

---

### User Story 4 - 過剰な複雑性のシンプル化 (Priority: P3)

機能面で必要以上に複雑になっている箇所が簡素化され、コードの可読性と保守性が向上している。

**Why this priority**: 他の優先事項に比べて影響範囲が局所的だが、長期的な保守コストを削減する。

**Independent Test**: 対象箇所のコード行数が削減されていること。機能テストが引き続きパスすること。

**Acceptance Scenarios**:

1. **Given** 水質フリーズの二重状態管理（`waterQualityFrozen` + `waterChangeOwnerId`）, **When** リファクタリング後に確認する, **Then** より簡潔なパターン（Set ベースのカウンター等）で同等の機能が実現されている
2. **Given** UI コンポーネント内の重複スタイル定義（Accordion の sx プロパティ等）, **When** リファクタリング後に確認する, **Then** テーマ定数として共有化されている

---

### Edge Cases

- リファクタリング後も既存の全機能（ポモドーロタイマー、魚の飼育、店舗購入、苔掃除等）が正常に動作すること
- VSCode globalState に保存された既存データとの後方互換性が維持されること（キー名変更時はマイグレーション関数で対応）
- esbuild バンドルが正常に生成され、拡張機能として正しくロードできること
- メッセージ type フィールドやコマンド ID を変更する場合、extension ↔ webview 間の全メッセージハンドラが整合していること

## In-Scope Refactoring Inventory

以下が本リファクタリングの具体的な変更対象である。これら以外の変更は out-of-scope とする。

### 重複排除・型統合
- `buildSpriteUriMap()` 関数の統合（providers/tank-panel.ts, companion-view.ts → 共有モジュール）
- `SpriteUriMap` 型の統合（4 ファイルの重複定義 → 単一エクスポート）
- `window.__SPRITE_URI_MAP__` の排除（3 ファイル → React Context Provider）
- Accordion sx スタイルの共有化（複数 webview コンポーネント → テーマ定数）

### 型の厳密化
- `any` 型の排除（game/state.ts の `migrateState` 等）
- 禁止パターン: `as any`, 明示的 `: any`, `<any>`
- 許容例外: VSCode API 等の型定義が提供されない外部 API のみ（`// eslint-disable-next-line` + 理由コメント必須）

### 定数化
- ゲームプレイ・ビジネスルール定数の集約（game/health.ts の閾値、game/deterioration.ts のレート等）
- 対象: ゲームメカニクスに影響する数値のみ。UI スペーシング、アニメーションタイミング、配列インデックス等は対象外

### 構造簡素化
- 水質フリーズの二重状態管理の簡素化（game/engine.ts）

### ディレクトリ構造の再編成
- ファイルの移動・統合・ディレクトリ名の変更を含む構造の論理的再編成
- 不要・未使用ファイルの削除

### 命名規則の統一
- 全ファイルの変数名・関数名・ファイル名の命名規則見直し
- globalState のキー名、メッセージ type フィールド、package.json のコマンド ID も必要に応じてリネーム可能（マイグレーション対応必須）

### リグレッションテスト
- カバレッジが不足しているフロー（タイマー、魚管理、店舗、苔掃除、永続化）のテスト追加

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: リファクタリング後、全ての既存機能が変更なく動作すること（機能的な変更を含まない）
- **FR-002**: `any` 型の使用を排除すること。禁止パターン: `as any`, `: any`, `<any>`。許容例外: VSCode API 等の外部 API のみ（eslint-disable + 理由コメント必須）
- **FR-003**: 重複していた関数・型定義が単一の共有モジュールに集約されていること
- **FR-004**: ゲームプレイ・ビジネスルールに関わるマジックナンバーが名前付き定数として管理されていること
- **FR-005**: グローバル変数（`window` オブジェクトへの直接代入）による状態受け渡しが排除され、React Context Provider で置換されていること
- **FR-006**: 全てのテスト（ユニット・統合）がパスすること
- **FR-007**: ESLint / Prettier によるコード品質チェックがパスすること
- **FR-008**: 不要・未使用ファイルが削除されていること
- **FR-009**: 公開契約（globalState キー、メッセージ type、コマンド ID）を変更する場合、マイグレーション関数で後方互換性を維持すること
- **FR-010**: カバレッジが不足しているフローのリグレッションテストが追加されていること

### Key Entities

- **GameState**: ゲームの全状態を保持するオブジェクト。永続化層（globalState）との接点。
- **GameStateSnapshot**: webview に送信される GameState の最適化された部分集合。
- **SpriteUriMap**: スプライト画像の URI マッピング。現在重複定義されており、単一の型に統合する対象。
- **StoreItem**: 購入可能アイテムの定義。タンク・フィルター・魚種の各レジストリと関連。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: コード内の `any` 型使用が 0 件（許容例外は eslint-disable + コメントで明示）
- **SC-002**: 重複関数・型定義が 0 件（同一定義が 2 箇所以上に存在しない）
- **SC-003**: 全ての既存テストがリファクタリング後もパスする
- **SC-004**: ESLint エラーが 0 件の状態を維持
- **SC-005**: 既存ユーザーの保存データ（globalState）が自動マイグレーションにより正常に読み込める
- **SC-006**: 主要フロー（タイマー、魚管理、店舗購入、苔掃除、永続化）のリグレッションテストが追加されている

## Clarifications

### Session 2026-03-28

- Q: リファクタリングの適用戦略は一括か段階的か？ → A: 一括適用（1 つの PR で全変更をまとめる）
- Q: リファクタリング後のパフォーマンス基準は？ → A: パフォーマンス基準なし（ビルド・テスト通過のみで十分）
- Q: リファクタリングのスコープ範囲は？ → A: 基本 7 項目に加え、ディレクトリ構造の再編成、全ファイルの命名規則見直し、不要ファイルの削除・統合を含む
- Q: `any` 型の禁止ルールは？ → A: `as any`, `: any`, `<any>` を禁止。VSCode API 等の外部 API のみ eslint-disable + コメント付きで許容
- Q: webview のデータフローパターンは？ → A: React Context Provider を採用
- Q: リグレッション検証レベルは？ → A: 既存テスト通過 + カバレッジ不足フローのテスト追加
- Q: 命名規則とリネームの境界は？ → A: 全てリネーム可能（globalState キー、メッセージ type、コマンド ID 含む）。ただしマイグレーション対応必須

## Assumptions

- 全変更を 1 つの PR にまとめて一括適用する（段階的分割は行わない）
- パフォーマンス基準は設けない（ビルド・テスト通過のみが合格条件）
- リファクタリングは機能変更を伴わない（振る舞いの変更なし）
- 状態管理の改善は React Context Provider を採用する
- パフォーマンスの最適化は本スコープ外（コード品質・可読性のみに注力）
- ファイル名の命名規則は kebab-case、export 名は camelCase/PascalCase を基本とする
- 公開契約（globalState キー、メッセージ type、コマンド ID）も必要に応じてリネーム可能だが、マイグレーション対応を必須とする
